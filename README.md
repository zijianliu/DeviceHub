# 设备借用与归还系统

一个完整的企业内部设备借用与归还管理系统，解决库存状态不同步、借出记录对不上、重复借出、逾期管理等实际问题。

## 技术栈

- **后端**: Node.js + Express + TypeScript + TypeORM + SQLite
- **前端**: React 18 + TypeScript + Vite + Ant Design
- **状态管理**: React Hooks (useState, useEffect)
- **路由**: React Router v6

## 核心特性

### 1. 设备管理
- 设备列表展示：设备编号、名称、分类、状态、位置、负责人
- 新增/编辑设备
- **设备编号唯一**约束（数据库唯一索引 + 代码校验）
- 设备详情页：完整信息、当前借用状态、历史借用记录、操作时间线

### 2. 借用流程
- 发起借用申请：借用人、借用时间、预计归还时间、用途说明
- 借用前校验：
  - 设备状态必须是"可借"才能借用
  - **同一设备不能存在未完成的借用记录**
  - 预计归还时间必须晚于借用时间
- 借用成功后：
  - 设备状态变为"借出中"
  - 生成借用记录
  - 记录操作日志
  - 列表页和详情页状态**同步更新**

### 3. 归还流程
- 发起归还：归还人、设备状态（完好/损坏）、备注
- 归还前校验：
  - 设备状态必须是"借出中"才能归还
  - 必须有对应的未完成借用记录
- 归还后：
  - 设备完好 → 状态变为"可借"
  - 设备损坏 → 状态变为"维修中"
  - 更新借用记录（实际归还时间、归还人、是否损坏、备注）
  - 记录操作日志

### 4. 逾期管理
- **动态计算逾期状态**：根据当前时间和预计归还时间判断
- 列表页明确标识逾期设备（红色标签）
- 详情页显示逾期天数
- 列表和详情状态**统一计算**，不会出现不一致

### 5. 状态流转规则
```
可借(available) → 借出中(lent), 维修中(maintenance), 停用中(disabled)
借出中(lent) → 可借(available), 维修中(maintenance)
维修中(maintenance) → 可借(available), 停用中(disabled)
停用中(disabled) → 维修中(maintenance)
```

### 6. 操作追溯
- 每台设备有完整的操作时间线
- 记录：创建、更新、借出、归还、维修、停用
- 包含：操作人、时间、状态变化、详细信息

### 7. 列表筛选
- 按分类筛选（笔记本、投影仪、测试机、相机、其他）
- 按状态筛选（可借、借出中、维修中、停用中）
- 按逾期状态筛选（已逾期/未逾期）
- 关键字搜索（设备名称或编号）
- 分页显示
- 翻页后筛选条件保留

## API 接口说明

### 设备相关 API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/health | 健康检查 |
| GET | /api/devices | 获取设备列表（支持分页、筛选） |
| GET | /api/devices/:id | 获取设备详情 |
| GET | /api/devices/categories | 获取设备分类列表 |
| GET | /api/devices/statuses | 获取设备状态列表 |
| POST | /api/devices | 创建设备 |
| PUT | /api/devices/:id | 更新设备 |
| GET | /api/devices/:id/borrow-records | 获取设备的借用记录 |

### 借用/归还相关 API

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/borrows/borrow | 发起借用 |
| POST | /api/borrows/return | 发起归还 |
| GET | /api/borrows/check-borrowable/:deviceId | 检查设备是否可借 |
| GET | /api/borrows/check-returnable/:deviceId | 检查设备是否可归还 |
| GET | /api/borrows/active/:deviceId | 获取设备的活跃借用记录 |

## 最小验证流程

### 验证 1：同一设备不能重复借出

**步骤：**
1. 打开设备列表，找到状态为"可借"的设备（如 MacBook Pro 14寸）
2. 点击"借用"按钮，填写借用人（如"测试用户"），提交
3. 确认借用成功后，设备状态变为"借出中"
4. 尝试再次点击"借用" - **按钮应该消失**，因为状态已不是"可借"
5. 打开浏览器开发者工具，手动尝试再次调用借用 API（使用同一 deviceId）
6. 后端应该返回错误：`"该设备已有未完成的借用记录"` 或 `"设备当前状态不可借用"`

**预期结果：**
- 借用后状态立即更新为"借出中"
- 无法再次借用同一设备
- 后端有双重校验（状态检查 + 活跃记录检查）

---

### 验证 2：归还后状态能恢复

**步骤：**
1. 找到状态为"借出中"的设备（如 ThinkPad X1 Carbon）
2. 点击"详情"查看当前状态和借用记录
3. 返回列表，点击"归还"按钮
4. 填写归还人（如"测试用户"），选择"设备状态：完好"
5. 提交归还

**预期结果：**
1. 归还成功后：
   - 设备状态从"借出中"变为"可借"
   - 借用记录的 `isReturned` 变为 `true`
   - 操作日志增加一条"归还"记录
2. 列表页和详情页状态**保持一致**

---

### 验证 3：逾期状态会动态变化

**步骤：**
1. 找到 Samsung Galaxy S24（TES-002），初始状态是"逾期"
2. 查看列表页：显示红色"逾期"标签
3. 点击"详情"：
   - 设备信息处显示"逾期 X 天"
   - 当前借用信息处也显示"已逾期 X 天"
   - 借用记录表格中也有逾期标识

**预期结果：**
1. 列表页、详情页、记录中的逾期状态**完全一致**
2. 逾期状态不是硬编码的，而是**根据当前时间动态计算**的
3. 如果修改系统时间到预计归还时间之前，逾期状态应该消失

---

### 验证 4：列表和详情状态保持一致

**步骤：**
1. 在列表页查看任意设备的状态
2. 点击"详情"进入详情页
3. 比较两处的状态显示

**预期结果：**
- 列表页和详情页的状态（含逾期状态）**完全一致**
- 状态数据源统一：都来自后端的 `isOverdue` 字段计算
- 没有各自独立计算导致的不一致

---

### 验证 5：借出和归还记录能对上

**步骤：**
1. 执行一次完整的借用流程：
   - 借一台设备（如 MacBook Pro 14寸）
   - 记录借用人：张三
   - 预计归还时间：明天

2. 查看详情页的借用记录：
   - 应该有一条新记录，状态为"借用中"
   - 借用人：张三
   - 实际归还时间：空

3. 执行归还流程：
   - 归还人：李四（或张三）
   - 设备状态：完好
   - 备注：设备完好归还

4. 再次查看借用记录：
   - 同一条记录的 `isReturned` 变为 `true`
   - 实际归还时间有值
   - 归还人有值
   - 备注有值

5. 查看操作时间线：
   - 有一条"借出"记录，操作人张三
   - 有一条"归还"记录，操作人李四
   - 两条记录对应同一次借用周期

**预期结果：**
1. 每次借用和归还都生成**对应的记录**
2. 借用和归还操作**关联到同一条记录**（不会创建新记录）
3. 操作时间线能完整追溯：借出时间 → 归还时间 → 操作人 → 状态变化
4. 通过借用记录和操作日志可以相互印证

## 核心约束实现

| 约束 | 实现位置 | 说明 |
|------|---------|------|
| 设备编号唯一 | `Device.code` 唯一索引 + `create()` 方法校验 | 数据库层面 + 代码层面双重保证 |
| 同一设备不能重复借出 | `canBorrow()` 方法 + `findActiveBorrowByDeviceId()` 查询 | 先检查状态，再检查是否有未完成的借用记录 |
| 状态流转规则 | `StatusTransitionRules` 常量 + `canTransitionStatus()` 方法 | 每种状态只允许切换到指定状态 |
| 逾期状态动态计算 | `isOverdue()` 方法 + 后端实时判断 | 根据当前时间和预计归还时间计算，不存储 |
| 操作追溯 | `OperationLog` 实体 + 每次操作都记录 | 每个状态变化都记录时间、操作人、详情 |
| 状态一致性 | 列表和详情使用相同的计算逻辑 | 都调用 `isOverdue()` 方法，结果一致 |

## 注意事项

1. **逾期状态**：逾期状态是动态计算的，不是存储在数据库中的字段。这确保了即使系统关闭后再打开，逾期状态也能正确显示。

2. **数据库同步**：首次启动后端会自动创建数据库表。`npm run seed` 只会在数据库为空时初始化数据。

3. **状态变更**：设备状态只能通过借用/归还/维修/停用操作变更，不能直接编辑状态字段。

4. **日志完整性**：每次状态变更都会记录操作日志，包括：
   - 操作类型（创建/更新/借出/归还/维修/停用）
   - 从什么状态变为什么状态
   - 操作人
   - 操作时间
   - 详细信息（JSON 格式）
